# MIREX-Anchored Emotion Classification Pipeline

**MoodSense · DSP 422 — Practical Machine Learning · Group 3**

This folder contains the complete non-circular emotion classification pipeline built and maintained by Albin Anto Jose, Ankit Mittal, Nandini Bag, and Kasheena Mulla from the Data Science Department at School of Professional Studies, Northwestern University. It addresses a fundamental methodological flaw found in the team's baseline approach and replaces it with a human-annotation-anchored system that produces genuine multimodal signal.

---

## The Problem This Solves

The team's primary dataset — [`devdope/900k-spotify`](https://www.kaggle.com/datasets/devdope/900k-spotify) (500K+ Spotify tracks) — includes emotion labels generated by [`mrm8488/t5-base-finetuned-emotion`](https://huggingface.co/mrm8488/t5-base-finetuned-emotion), a T5-based NLP model that infers emotion from **lyrics text**. The dataset also contains over 30 audio and contextual features (Energy, Danceability, Tempo, Loudness, Valence, etc.).

The baseline model trained on these labels achieved 61–65% accuracy — but the result is **circular by construction** for any lyrics-based approach: the T5 model assigned labels by processing the same lyrics text used as training features. Predicting NLP-derived labels from the same lyrics is tautological, equivalent to learning an identity mapping.

Crucially, this circularity is confirmed empirically via the **audio feature control**: the dataset has rich audio features that are completely independent of lyrics. Adding those audio features to a TF-IDF lyrics model gave **exactly 0 percentage points of lift**. If the labels encoded genuine perceptual mood, audio features would contribute something. The fact that they contribute nothing reveals that the T5-derived label space captures linguistic style, not the perceptual mood that audio encodes.

The MIREX pipeline breaks this by anchoring labels to **human perception** — 903 audio clips and 764 lyrics files from the [Multi-modal MIREX-like Emotion Dataset](https://mir.dei.uc.pt/downloads.html) (Panda et al., CMMR 2013), annotated by human listeners. Under human-annotated labels, audio features contribute a genuine **+3.28 pp lift**, confirming the label source — not the model or feature set — was the bottleneck.

## How Labels Were Generated (Training)

Standard emotion datasets either use NLP-derived labels (circular with lyrics — the T5 label case above) or synthetic audio-rule labels (circular with audio). MIREX avoids both by using direct human perceptual ratings of audio clips.

**Step-by-step:**

1. **MIREX anchor set** — 903 audio clips (30s each) and 764 lyrics files from the [Multi-modal MIREX-like Emotion Dataset](https://mir.dei.uc.pt/downloads.html) (Panda et al., CMMR 2013), annotated by human listeners into 5 perceptual clusters. Of the 903 clips, 764 had matching lyrics — only songs with both modalities were used for lyric-based centroid computation.

2. **Cluster → emotion mapping** (by inspecting cluster centroids and representative tracks):

   | MIREX Cluster | Emotion |
   |---|---|
   | Cluster 1 | Love |
   | Clusters 2 + 4 | Happy |
   | Cluster 3 | Sad |
   | Cluster 5 | Anger |

3. **Lyric embedding of anchor songs** — Each MIREX song's lyrics were embedded with OpenAI `text-embedding-3-small` (512d for centroid computation during label assignment). Per-emotion **centroids** were computed in the 512d text embedding space. Note: the final trained model and inference pipeline use **1536d** embeddings — the higher dimension was used for all feature vectors in training and prediction, with 512d used only for the label propagation step.

4. **Label propagation to 120K songs** — A separate [`serkantysz/550k-spotify-songs-audio-lyrics-and-genres`](https://www.kaggle.com/datasets/serkantysz/550k-spotify-songs-audio-lyrics-and-genres?select=songs.csv) dataset was used as the training base. This dataset has no pre-assigned emotion labels, making it a clean slate for MIREX-anchored labelling. It was genre-balanced to ~120K songs. Each song's lyrics were embedded at 512d and the **nearest MIREX centroid** (cosine similarity) assigned the emotion label.

5. **Rebalancing** — The raw label distribution was Sad-heavy (46%). The dataset was downsampled to **70,568 songs** (17,642 per class) for class balance.

### Full Ablation Results

| Model | Features | Dimensions | Accuracy | Macro F1 |
|---|---|---|---|---|
| LinearSVC | Text only | 512d | 59.95% | — |
| LinearSVC | Text + Audio | 524d | 62.38% | 0.62 |
| LightGBM (default) | Text + Audio | 524d | 65.67% | 0.66 |
| LightGBM (tuned) | Text + Audio | 524d | 66.95% | 0.67 |
| **LightGBM (tuned)** | **Text + Audio** | **1548d** | **67.18%** | **0.67** |

The **audio lift under MIREX labels** (LinearSVC: +2.43 pp, LightGBM: +3.28 pp) versus **0 pp lift under T5-derived NLP labels** is the central quantitative finding of this work.

### Per-Class F1 (Best Model)

| Emotion | F1 |
|---|---|
| Love | 0.73 |
| Anger | 0.68 |
| Happy | 0.66 |
| Sad | 0.63 |

### Hyperparameters (Tuned LightGBM)

A manually selected set of well-performing values was tested across `n_estimators`, `max_depth`, `learning_rate`, `num_leaves`, and `min_child_samples`. Exhaustive search methods such as GridSearchCV or Optuna were not used due to computational constraints — these are recommended as a future step for a potential small additional lift in accuracy. Final model: `lgbm_mirex_1548d.pkl` (~27 MB).

## Usage

### Python API

```python
from mirex_pipeline.inference import classify_song

# Minimal call
result = classify_song("Blinding Lights")

# With artist for more accurate Spotify search
result = classify_song("Blinding Lights", artist="The Weeknd")

print(result)
# {
#   'emotion':       'Happy',
#   'confidence':    0.7231,
#   'probabilities': {'Anger': 0.0412, 'Happy': 0.7231, 'Love': 0.1104, 'Sad': 0.1253},
#   'track':         {'id': '...', 'name': 'Blinding Lights', 'artist': 'The Weeknd',
#                     'embed_url': 'https://open.spotify.com/embed/track/...'},
#   'lyrics_found':  True
# }
```

### Return Schema

| Key | Type | Description |
|---|---|---|
| `emotion` | `str` | Predicted class: `Happy`, `Sad`, `Anger`, or `Love` |
| `confidence` | `float` | Probability of the predicted class |
| `probabilities` | `dict` | Softmax probabilities for all 4 classes |
| `track` | `dict` | Spotify track metadata + embed URL |
| `lyrics_found` | `bool` | `False` if Genius returned no lyrics (falls back to track name embedding) |

### Fallback Behaviour

If lyrics are not found on Genius, the song **title string** is embedded instead of lyrics. Classification still runs but accuracy may be lower. The `lyrics_found` flag allows callers to surface this to the user.

## Key Design Decisions & Limitations

**Why OpenAI embeddings instead of Sentence Transformers?**
`all-MiniLM-L6-v2` (384d) was tested and reached 52% — worse than TF-IDF (61%). `text-embedding-3-small` at 1536d was chosen for representation quality. Cost per inference is negligible (<$0.001/song).

**Why ReccoBeats instead of Spotify Audio Features?**
Spotify deprecated its Audio Features endpoint on November 27, 2024. ReccoBeats provides an equivalent set of features (danceability, energy, valence, tempo, etc.) derived from the same Spotify track IDs.

**Why 4 classes instead of 5?**
MIREX Clusters 2 and 4 were merged into `Happy` after inspecting representative tracks and noting their centroid proximity in embedding space. The `Love` class has the fewest anchor samples in the MIREX set, which explains its relatively lower recall in some ablation runs despite achieving the highest F1 overall (0.73) in the final model.

**Performance ceiling**
The 67.18% ceiling is bounded by MIREX anchor quality (903 clips, 764 with lyrics, unequal cluster sizes, genre skew toward Western pop). Expanding the anchor set with a more diverse annotated corpus would be the highest-leverage improvement.
